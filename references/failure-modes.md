# Failure modes — how learners fool themselves

Read this when diagnosing why a user is stuck, spinning, or reporting progress that doesn't match their actual ability. Each entry: the pattern, the detection cue, the correction.

## 1. Fluency illusion

**Pattern:** User feels they understand because an explanation made sense in real time. They conflate "I followed it" with "I know it."

**Detection cues:**
- "Yeah that makes sense, what's next"
- "I get the idea" after a single exposure
- Can paraphrase the explanation but can't produce it from a prompt without it
- Asks you to re-explain rather than attempting a problem

**Correction:** Pull the explanation away. Give a problem that requires retrieval of the specific thing just taught, in a context where it isn't primed. If they can't do it, the "understanding" was WM, not LTM. Name this when it happens.

## 2. Reference crutch

**Pattern:** User "studies" by solving problems with the textbook, notes, or prior worked examples open. Information stays in WM for the session and dissipates after.

**Detection cues:**
- Reports doing lots of problems but can't recall the formulas, theorems, or patterns from memory
- Says "I just had this yesterday" but can't reproduce it
- Looking things up feels like a neutral act to them, not a failed rep

**Correction:** Closed-book retrieval first, every time. Reference is a spotter — used only when genuinely stuck, one cue only, then try to finish unaided. Any rep done with the reference open is a consumption rep, not a retrieval rep. Count them separately.

## 3. Shortcut-to-advanced

**Pattern:** User wants to start with the cool advanced topic (transformers, PDEs, compilers) and fill in prereqs as needed. Hits a wall, concludes they're not smart enough.

**Detection cues:**
- Target skill far from their current solid ground
- Questions jump between layers ("what's a gradient?" mid-backprop)
- Frustration spiking, reports of "just not getting it"
- Roadmap they've built has gaps they can't see

**Correction:** Top-down plan, bottom-up execution. Map the real prereq chain. Identify the deepest unsolid node. Start there even if it feels like regression. The train has to start moving; advanced flailing is backward motion.

## 4. Projects-as-fundamentals

**Pattern:** User believes they learn best "by building." Tries to acquire foundational skills through project work. Project either stalls or produces a working artifact without the underlying skill being in LTM.

**Detection cues:**
- "I learn best by doing"
- Has built several projects but can't reproduce the core techniques on demand
- Project complexity keeps hitting WM ceiling
- Each project teaches 20% of what it should because cognitive load is all bookkeeping

**Correction:** Projects are a capstone, not a curriculum. Pull the fundamental back out into its own scaffolded practice. Build it in isolation to mastery. Then return to project work where it now costs near-zero WM to deploy.

## 5. Completionism without retrieval

**Pattern:** User works through a book, course, or problem set linearly and "completes" it. Measures progress in pages, videos watched, problems attempted.

**Detection cues:**
- Tracks pages/hours/problems-attempted, not retention
- "I finished Strang" or "I did MIT 6.006" as if that's the endpoint
- Can't produce content from chapter 3 a month later
- Review happens (if at all) by re-reading

**Correction:** Replace completion metrics with retention metrics. Problems solved unassisted at difficulty D, a week after first exposure. Percent recall of a spaced review deck. Actual skill the user can demonstrate on demand. "Finished" is not a learning metric.

## 6. Bloated feedback loop

**Pattern:** User's practice ritual has so much ceremony that one rep takes forever. Note-taking perfectionism, overly elaborate spaced-rep decks, long pre-planning, think-pair-share dynamics. Result: 3 reps in an hour when the session should have 30.

**Detection cues:**
- Studied for 2 hours, can recount only a handful of problems attempted
- Spends more time organizing study materials than studying
- Each problem includes a long narration, self-reflection, and writeup
- Progress is slow on material that should be moving

**Correction:** Strip the loop. Problem → attempt → check → next. Volume matters. Mindful ≠ slow. Deliberate practice is tight feedback, not ceremonial practice.

## 7. Wrong-depth difficulty

**Pattern:** User keeps picking problems either far too easy (comfortable, no gain) or far too hard (flailing, no successful reps). Either way, little consolidation.

**Detection cues:**
- "That was easy, what's next" repeatedly → too easy
- Gets zero unassisted reps successful, frustration rising → too hard
- Jumps between trivial and impossible with no stable zone

**Correction:** Find the zone where reps are challenging but successful 60–80% of the time. Drop complexity (strip context) when flailing; add context layers when cruising. Review in particular should feel as taxing as initial learning — if it's easy, the gap was too short.

## 8. Skipping the "tedious" rep

**Pattern:** User knows the concept intellectually, refuses to drill the computational/mechanical fluency. Thinks drilling is beneath them.

**Detection cues:**
- "I understand the idea, I don't need to practice the computation"
- Makes "silly mistakes" consistently on execution
- Slow on anything requiring mechanical application even when the concept is clear
- Treats practice as unsophisticated

**Correction:** Silly mistakes = insufficient practice. Intellectual understanding without automatized execution means the mechanics are eating WM during harder problems. Drill the boring layer until it costs nothing. This is where much of the real alpha is.

## 9. Overreacting to bad sessions

**Pattern:** User has a rough day, couldn't retrieve stuff they thought they knew, concludes the method isn't working or they're not cut out for it.

**Detection cues:**
- Dramatic swings in self-assessment
- "Maybe I don't have what it takes"
- Wants to restructure the whole plan after one bad day
- Confuses day-to-day variance with trajectory

**Correction:** Individual sessions are noisy. Trajectory is the signal. Hold the method, protect the habit, don't renegotiate the plan based on one data point. If the trend over weeks is bad, *then* revisit.

## 10. Asking-to-be-told vs. attempting

**Pattern:** When faced with a question, user defaults to asking for the answer or the method before attempting. Retrieval never happens because they never try.

**Detection cues:**
- "How do I solve this" before spending any effort on it
- Wants the approach explained before attempting
- Tries to extract hints instead of reps

**Correction:** "Attempt first. I'll grade, not pre-explain." Even a failed attempt makes the subsequent explanation stick dramatically better than pre-explanation, because the failure primes the relevant neurons. No attempt → no retrieval → no gain.
